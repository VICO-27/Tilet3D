# 06. Orders System

---

# Orders Module

Location:

```
apps/orders/
```

The Orders module is responsible for converting a customer's shopping cart into a permanent purchase record.

Unlike the Cart module, which is temporary and editable, an Order represents a finalized purchase that becomes part of the business records.

The Orders app also coordinates with the Inventory and Payments systems to ensure that products cannot be oversold and that payment processing remains consistent.

---

# Responsibilities

The Orders module is responsible for:

- Creating orders from shopping carts
- Creating immutable order snapshots
- Reserving inventory during checkout
- Managing order status
- Managing payment status
- Tracking shipping information
- Calculating totals
- Preventing duplicate checkout
- Providing order history

---

# Checkout Service

Location

```
apps/orders/services/checkout.py
```

This service is the single entry point for checkout.

Nothing else in the project should create orders directly.

Responsibilities include:

- Loading the cart
- Locking inventory
- Validating stock
- Reserving stock
- Creating the order
- Creating order items
- Clearing the cart
- Creating the payment
- Initializing the payment gateway

Everything runs inside one database transaction.

```
@transaction.atomic
```

This guarantees that checkout either completes entirely or rolls back completely.

---

# Cart Validation

Checkout begins by loading the authenticated user's cart.

Example

```
Cart.objects
    .select_for_update()
    .prefetch_related(...)
```

Using

```
select_for_update()
```

prevents two checkout requests from modifying the same cart simultaneously.

If the cart is empty:

```
Your cart is empty.
```

checkout immediately stops.

---

# Inventory Reservation

Before any order is created, every cart item is validated.

Each product variant must satisfy:

- Active product
- Available inventory
- Enough stock

Reservation occurs before payment.

Example

```
InventoryService.reserve(...)
```

This moves inventory from

Available Stock

↓

Reserved Stock

instead of immediately reducing inventory.

This prevents overselling.

Every order receives a human-readable identifier.

Example

```
TLT-20260709-000001
```

Generated by

```
OrderNumberService
```

Format

```
TLT

YYYYMMDD

Daily Sequence
```

Examples

TLT-20260709-000001

TLT-20260709-000002

- Easy warehouse lookup

# Order Items

After the order is created, each cart item becomes an OrderItem.

Example

```
Cart Item

↓

Order Item
```

Each OrderItem stores a snapshot of:

- Product ID
- Variant ID
- Product Name
- Variant Name
- SKU
- Color
- Size
- Price
- Quantity
- Subtotal

This protects historical accuracy.

---

# Snapshot Strategy

The system intentionally copies product information into the order.

Example

Today

```
Habesha Kemis

3500 ETB
```

Tomorrow

Admin changes price

```
4200 ETB
```

Old order still shows

```
3500 ETB
```

instead of

```
4200 ETB
```

This preserves financial history.

---

# Why Snapshots Matter

Without snapshots

```
Order

↓

Product

↓

Current Price
```

Historical orders would change every time the product changes.

This is unacceptable.

Snapshots guarantee:

- auditability
- accounting accuracy
- customer invoice consistency

---

# Clearing the Cart

Only after the order has been successfully created are cart items removed.

```
cart.items.all().delete()
```

This prevents duplicate checkout.

# Order Status Lifecycle

Orders move through predefined states.

```
Pending

↓

Confirmed

↓

Processing

↓

Shipped

↓

Delivered
```

Possible cancellation

```
Pending

↓

Cancelled
```

or

```
Confirmed

↓

Cancelled
```

or

```
Processing

↓

Cancelled
```

Delivered orders cannot be cancelled.

---

# Order Lifecycle Service

Location

```
apps/orders/services/lifecycle.py
```

This service acts as a state machine.

Instead of allowing code to modify order status directly

```
order.status = ...
```

all transitions pass through

```
OrderLifecycle.transition(...)
```

This prevents invalid transitions.

---

# Allowed Transitions

Current Status | Allowed Next Status
-------------- | -------------------
Pending | Confirmed, Cancelled
Confirmed | Processing, Cancelled
Processing | Shipped, Cancelled
Shipped | Delivered
Delivered | None
Cancelled | None

---

# Invalid Transition Example

Attempting

```
Delivered

↓

Pending
```

raises

```
OrderLifecycleError
```

This guarantees business consistency.

# Payment Status

Orders track payment separately.

Possible values include:

- Pending
- Paid
- Failed

This separation allows scenarios such as:

```
Order

Confirmed

Payment

Pending
```

or

```
Order

Cancelled

Payment

Refunded
```

# Concurrency Protection

Checkout uses:

```
select_for_update()
```

and

```
transaction.atomic()
```

to ensure:

- one checkout per cart
- no duplicate orders
- no race conditions

# Design Decisions

The Orders module follows several important architectural decisions.

### Service Layer

Business logic never lives inside API views.

---

### Snapshot Pattern

Orders never depend on mutable product data.

---

### Transactional Checkout

Checkout is fully atomic.

---

### Inventory Reservation

Inventory is reserved before payment and finalized after successful payment.

---

### Human-Friendly Order Numbers

UUIDs are excellent database identifiers.

Customers prefer readable order numbers.

The system uses both.
